# 邻居，你好！(Neighbor, Hello!) v1 需求与技术规格（交付给代码AI用）

> 版本：v1（基于当前对话确认的全部决策）  
> 适用范围：小区（Community）级别的邻里协作网站  
> Slogan：**远亲不如近邻！**  
> 目标：以“家庭”为信任与治理单位，提供邻里互助板块与**去中心化社区自治治理**（无平台管理员裁决）。

---

## 0. 本文档目的与最终交付物（你最初目标）

你要实现一个网站：**邻居，你好！**，提供：

1. **账号注册**（不强制手机号，但可选绑定）  
2. **创建/加入家庭**：家庭可添加用户，用户必须设置角色（男主人、女主人、孩子1-3、长辈1-3）。  
3. **以小区为单位的邻里关系**：家庭与家庭互加好友成为“好邻居”。  
4. **v1 系统自带板块**（上线即有，不依赖后台配置创建）：  
   - 约饭  
   - 帮带娃  
   - 帮遛狗  
   - 上班拼车  
   - 一起运动  
   > v1 暂不包含“众筹小区设施”（已确认：先不要）。
5. **去中心化自治治理（核心）**：  
   - 无平台管理员裁决；所有治理通过全体参与者投票执行（按家庭治理、家庭管理员代表投票）。  
   - 内容隐藏、禁言、解禁、参数调整等治理行为都复用统一“提案-投票-生效”框架。  
   - 投票支持：同意 / 反对 / 弃权；倒计时结束不投票算弃权。  
   - 通过需满足：最低参与数 + 同意占基数达标 + 反对占基数不超过上限。  
   - **A 规则**：反对票达到否决阈值后**提前结束并判失败**。  
   - 被隐藏累计达到阈值触发禁言（递进时长），可申诉解禁；解禁按活跃家庭计算；被禁言用户不能投票（包括自己的解禁提案）。  
6. 可运营、可扩展：治理阈值、投票周期、处罚阶梯等均通过治理参数配置并可投票变更；并支持按小区规模自动切档（20/50/100/300/500/1000）。

**最终要交付的产品形态**（对实现AI的明确要求）：  
- 一套可运行的后端服务（FastAPI + PostgreSQL），提供完整 API（含鉴权、家庭/小区、板块内容、治理提案与投票、通知）。  
- 数据库模型可落地（迁移/建表脚本）。  
- 系统启动后自动 seed 系统板块。  
- 治理参数 config 默认值齐全并能随小区规模切档。  
- 关键流程闭环：创建家庭加入小区 → 发帖/互动 → 提案投票隐藏 → 累计触发禁言 → 申诉解禁。

## 2. 核心概念与规则（产品层）

### 2.1 关键实体
- **用户（User）**：账号主体；可能属于一个家庭；可能处于正常/禁言/封禁状态  
- **家庭（Family）**：信任与治理的最小单元；家庭内有成员、角色、管理员  
- **小区（Community）**：治理与内容的空间边界；家庭加入小区后成为“全体家庭”成员  
- **好友关系（Neighbor Friendship）**：家庭对家庭互加好友  
- **板块（Board）**：系统内置 5 个；帖子按板块归类  
- **帖子/评论（Post/Comment）**：内容载体；可被治理隐藏/恢复  
- **参与/报名（Participation）**：帖子（约饭/拼车等）可报名参与  
- **治理提案（Proposal）/投票（Vote）**：统一治理框架

### 2.2 家庭角色（v1 固定）
角色 code 建议：
- `MALE_HOST` 男主人
- `FEMALE_HOST` 女主人
- `CHILD_1/2/3` 孩子1/2/3
- `ELDER_1/2/3` 长辈1/2/3

> 角色是“成员属性”，与权限无强绑定（治理权只与是否为家庭管理员有关）。

### 2.3 家庭管理员（House Admin）
- 每个家庭有且只有 1 个管理员
- **只有家庭管理员可代表家庭参与治理投票、发起治理提案**
- 管理员可在家庭内部转让（需要被转让人确认）
- 管理员被禁言或封禁时：系统自动切换管理员至其他未封禁成员
- 被封禁账号永久不得再担任管理员

### 2.4 新家庭治理冷却期
家庭加入小区后：**3 天内无治理权**（不可发起提案、不可投票）。

### 2.5 注册不强制手机号（安全底线）
由于不强制手机号，v1 必须具备：
- 创建家庭/兑换邀请码接口 **图形验证码 + IP/账号限流**
- 新家庭 3 天治理冷却（已定）

## 3. 用户主流程（关键闭环）

### 3.1 加入小区（邀请码）并创建家庭
1. 用户注册/登录（手机号可选）
2. 输入小区邀请码（仅用于“创建家庭加入小区”）
3. 通过图形验证码
4. 创建家庭（家庭名等）
5. 家庭加入小区；治理权在 `joined_at + 3天` 生效

### 3.2 加入已有家庭（不消耗小区邀请码）
1. 用户注册/登录
2. 通过家庭邀请链接/码加入家庭
3. 选择角色
4. 继承家庭所在小区与权限（但个人治理权仍取决于是否为家庭管理员）

### 3.3 发帖与参与
1. 家庭成员在板块发帖（约饭/带娃/遛狗/拼车/运动）
2. 其他成员评论、报名参与
3. 可见性：小区可见 / 仅好友可见（v1 支持）

### 3.4 治理闭环：隐藏 → 累计触发禁言 → 申诉解禁
1. 家庭管理员发起 `HIDE_POST/HIDE_COMMENT` 提案  
2. 各家庭管理员投票（YES/NO/ABSTAIN；不投=ABSTAIN）  
3. 满足通过条件则隐藏；作者**立即收到理由与投票结果**（透明）  
4. 若某账号在 360 天窗口内被隐藏通过次数累计 ≥ 3 → 触发禁言（递进时长）  
5. 被禁言用户可发起 `UNMUTE_USER` 解禁提案（但不能投票）  
6. 解禁投票按**活跃家庭**基数计算，通过则解禁

## 4. 社区自治治理协议（核心）

### 4.1 计票主体
- **按家庭治理**：每家庭 1 票
- **只有家庭管理员投票**

### 4.2 投票选项与默认行为
- 选项：同意（YES）/反对（NO）/弃权（ABSTAIN）
- 倒计时结束未投票：**自动记为弃权**

### 4.3 投票通过条件（统一公式）
设：
- `N_total`：投票基数（见 4.4）
- `A`：YES 票数，`O`：NO 票数，`P`：ABSTAIN 票数（含自动弃权）
- `T = A + O + P`：参与数

提案 **通过** 需同时满足：
1. `T >= minVotes`（最低参与数）
2. `A / N_total >= passYesRatio`（同意占基数比例达标）
3. `O / N_total <= vetoNoRatio`（反对占基数比例不超过上限）

### 4.4 基数定义：全体事项 vs 日常事项
#### A) 全体事项（global）
- `N_total = 全体家庭数`  
- 用于：治理参数变更、重大规则变更等整体决策

#### B) 日常事项（daily）
以活跃家庭为主，并加入“基数下限”避免小圈子劫持：
- `N_active = 活跃家庭数（30天活跃，且已过3天冷却）`
- `N_total = max(N_active, ceil(全体家庭数 * 20%))`  
其中 `20%` 作为治理参数可配置（v1 默认 0.20）。

### 4.5 A 规则：反对达否决阈值提前结束
投票进行中，若出现：
- `O / N_total >= vetoNoRatio`

则系统：
- 立即将提案状态置为 `FAILED_EARLY_VETO`  
- 提前结束投票并通知相关人员

### 4.6 透明度（作者透明通知）
当内容被隐藏/恢复、用户被禁言/解禁时：
必须向当事人立即展示/通知：
- 提案理由（reason）
- 票数分布（YES/NO/ABSTAIN）
- 基数 `N_total` 与阈值（minVotes、passYesRatio、vetoNoRatio）
- 提案 ID、截止时间、最终状态

## 5. 治理提案类型（v1）

### 5.1 内容治理
1) `HIDE_POST` / `HIDE_COMMENT`（scope: daily）  
- 通过后：目标内容 `status=hidden`；记录 `hidden_by_proposal_id`  
- 作者立即收到透明通知  

2) `RESTORE_POST` / `RESTORE_COMMENT`（scope: daily）  
- 通过后：目标内容恢复展示 `status=normal`  
- 作者立即收到透明通知  

### 5.2 用户治理
3) 自动禁言（不一定表现为提案类型；实现为系统自动动作）  
- 触发：360天窗口内“隐藏通过”累计 ≥ 3  
- 执行：按递进阶梯设置 `mute_until`  
- 通知：当事人（透明列出触发依据）

4) `UNMUTE_USER`（scope: daily；按活跃家庭）  
- 发起人：被禁言用户本人（允许发起但不允许投票）  
- 投票人：家庭管理员  
- 通过后：清除禁言  

> v1 不强制实现 `BAN_USER` 永久封禁提案；若要实现建议属于 global。

### 5.3 参数治理
5) `CHANGE_GOVERNANCE_PARAMS`（scope: global）  
- 修改 governance config（见第 8 节）并生成新版本，按 `effective_at` 生效  
- 生效时全体公告  

---

## 6. 处罚规则（v1 默认）

### 6.1 触发规则
- 统计窗口：`windowDays = 360`
- 当某用户在窗口内“其发布内容被隐藏通过”的次数累计 `>= 3` → 触发禁言

### 6.2 禁言递进时长（默认，可参数化）
- 第 1 次：24 小时
- 第 2 次：7 天
- 第 3 次：30 天
- 第 4 次：180 天
- 第 5 次：进入“永久禁言/封禁提案”流程（v1 可先只做永久禁言或提示后续）

### 6.3 禁言期间权限
- 禁言用户：不可发帖/评论/发起提案/投票
- 例外：**允许发起 `UNMUTE_USER` 解禁提案**，但仍不可投票（包括自己的解禁提案）

## 7. 小区规模切档（20/50/100/300/500/1000）与公告

### 7.1 切档规则
按 `全体家庭数` 选择当前档位：
- 取最小的 `stage >= all_family_count` 作为当前档  
例：37 户 → 50 档；120 户 → 300 档。

### 7.2 升级公告与生效
当全体家庭数突破当前档位上限进入下一档：
- 系统向全体家庭发送公告（站内通知）
- 新参数在 `applyAfterHours=48` 小时后自动生效
- **进行中的提案按创建时 snapshot 的参数结算**（避免中途改规则争议）

## 9. 技术方案（后端：FastAPI + PostgreSQL）

### 9.1 技术栈（建议）
- FastAPI + Uvicorn
- PostgreSQL
- SQLAlchemy 2.x（async） + Alembic
- Redis（可选但推荐：验证码存储/限流/通知队列/缓存）
- JWT（access+refresh）
- 图形验证码：服务端生成（PNG/base64）+ 一次性 token

### 9.2 服务结构建议
```text
app/
  main.py
  core/          # config、security(JWT)、rate_limit、errors、time utils
  db/            # async session、base、alembic
  models/        # ORM models
  schemas/       # Pydantic schemas
  services/      # governance eval、mute escalation、admin switch、seed
  api/v1/        # routers
  scripts/       # seed、maintenance jobs
```

### 9.3 工程实现要点（必须）
1. **投票事务一致性**：`vote -> 聚合 -> 判定 -> 状态更新 -> 执行动作` 必须在事务里完成，避免并发双写/重复执行。  
2. **A 提前否决**：每次投票后立即检查 `NO/baseN >= vetoNoRatio`，若达标立刻失败并关闭。  
3. **一家庭一票**：`votes` 表对 `(proposal_id, family_id)` 做唯一约束。  
4. **权限**：  
   - 发起提案/投票：必须是家庭管理员 + 家庭治理已生效 + 用户未禁言/封禁  
   - 禁言用户：只能发起 `UNMUTE_USER`，不能投票  
5. **限流**：创建家庭/兑换邀请码必须限流（IP/账号维度），否则邀请码泄露将被刷家庭。  
6. **系统板块 seed**：启动/部署必须 upsert 5 个板块；不可删除（`is_system=true`）。  
7. **公告**：切档与参数变更必须创建公告通知（notifications），并可在前端“公告栏”展示。

## 10. 数据模型（数据库表草图）

### 10.1 用户/家庭/小区

### 11.9 Governance（提案与投票）

#### POST `/api/v1/proposals`
家庭管理员发起治理提案（禁言用户不可发起；新家庭冷却期内不可发起）。
- Request（示例：隐藏帖子）
```json
{
  "communityId": "cm_1",
  "type": "HIDE_POST",
  "target": { "type": "post", "id": "p_1" },
  "reason": "包含不适宜内容",
  "evidence": { "note": "截图见附件", "images": ["file_1"] }
}
```
- Response
```json
{ "proposalId": "gp_1", "deadlineAt": "2026-04-10T00:00:00Z", "snapshot": { "...": "..." } }
```

#### POST `/api/v1/proposals/unmute`
禁言用户专用：发起解禁提案（仅此一种允许）。
- Request
```json
{ "communityId": "cm_1", "reason": "我已整改并保证遵守规则", "evidence": {} }
```
- Response `{ "proposalId": "gp_9" }`

#### GET `/api/v1/proposals`
列表（可筛选：type/status/target/createdBy）。

#### GET `/api/v1/proposals/{proposalId}`
详情需展示透明信息：票数、基数、阈值快照、倒计时、状态等。

#### POST `/api/v1/proposals/{proposalId}/vote`
家庭管理员投票。
- Request
```json
{ "choice": "YES" }
```
- Response（返回最新票数与状态，便于前端实时展示）
```json
{
  "status": "VOTING",
  "counts": { "yes": 3, "no": 1, "abstain": 5 },
  "baseN": 28,
  "thresholds": { "minVotes": 10, "passYesRatio": 0.25, "vetoNoRatio": 0.20 }
}
```

##### 投票接口关键业务规则（必须实现）
- 校验投票者为该家庭当前管理员
- 校验用户未禁言/封禁
- 校验家庭治理已生效（加入后 3 天）
- 一家庭一票（upsert + 唯一约束）
- 投票后立即判定：
  1) 若 `no/baseN >= vetoNoRatio` → `FAILED_EARLY_VETO`（提前结束）
  2) 否则若满足通过条件 → `PASSED`
  3) 否则保持 `VOTING`
- 状态变更后触发动作（隐藏/恢复/解禁/参数发布等）与通知

## 13. 安全与反滥用（v1 最小集合）
1. 创建家庭/兑换邀请码：图形验证码 + 限流（IP/账号维度）  
2. 新家庭治理冷却期：3 天  
3. 投票/提案接口事务一致性：防重复执行  
4. 审计与可追溯：`proposal.snapshot_json` + `moderation_events`  

---

## 14. 仍可选但推荐的实现细节（不改变需求，提升稳定性）
- 活跃家庭统计缓存（每日刷新），避免每次投票都重算大查询  
- 投票到期结算定时任务（cron/worker）：把 `deadline_at < now` 仍为 `VOTING` 的提案结算为 `PASSED/FAILED/EXPIRED`  
- 通知异步化（队列/后台任务），但 v1 也可同步写表

## 15. v1 验收清单（交付给实现AI）
1. 系统启动后 `boards` 表自动存在 5 个系统板块  
2. 可创建邀请码（后台）并设定有效期与次数，可撤销  
3. 用户注册后可通过邀请码+图形验证码创建家庭加入小区  
4. 家庭可邀请成员加入并设置角色  
5. 家庭可申请与其他家庭成为好友  
6. 五个板块均可发帖、评论、报名参与，且支持“小区可见/仅好友可见”  
7. 家庭管理员可发起隐藏/恢复提案并投票；投票支持 YES/NO/ABSTAIN，不投=ABSTAIN  
8. 提案通过条件实现：minVotes + passYesRatio + vetoNoRatio  
9. **A 规则**实现：NO 达 vetoNoRatio 立即提前失败并结束  
10. 内容被隐藏后作者立即收到透明通知（理由+票数+阈值+基数）  
11. 360 天窗口内隐藏通过累计 ≥3 自动触发禁言；禁言时长递进  
12. 禁言用户不可投票、不可发起提案，唯独可发起解禁提案但不能投票  
13. 解禁提案按活跃家庭基数结算并可解禁  
14. 小区规模突破档位时发送公告并 48h 后切换新档参数；进行中提案按快照结算
